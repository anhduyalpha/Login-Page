- Entry point `src/main.jsx` mounts `<App />` inside `ReactDOM.createRoot` and imports global styles from `src/styles/index.css`.
- `src/App.jsx` is the root component: it wraps everything in `AuthProvider`, renders an `AuthLayout` shell, and uses `framer-motion`'s `AnimatePresence` to switch between three views (`register`, `login`, `profile`) based on the `currentView` state; motion variants are reduced when `useReducedMotion()` is true.
- `src/context/AuthContext.jsx` owns all auth state (`currentUser`, `loading`, `currentView`, `toastNotification`) and exposes `register`, `login`, `logout`, `updateProfile`, `navigateTo`, plus a toast bus via `showToast`/`clearToast`; it subscribes to Firebase's `onAuthStateChanged` and seeds initial state from `getCurrentAuthUser()`.
- `src/services/firebase.js` is the sole persistence layer: it conditionally initializes Firebase (`initializeApp`, `getAuth`, `getFirestore`) only when `VITE_FIREBASE_*` env vars are present and not a demo key, otherwise falls back to a localStorage-based engine (`aether_auth_demo_users`, `aether_auth_current_user`). All public functions (`registerUser`, `loginUser`, `logoutUser`, `updateUserProfileData`, `getCurrentAuthUser`) share this dual-backend contract so callers never branch on backend choice.
- `src/pages/{LoginPage,RegisterPage,ProfilePage}.jsx` are dumb view components that consume `useAuth()` for actions and render through shared UI primitives under `src/components/` (`AuthCard`, `FormField`, `PrimaryButton`, `PasswordMeter`, `AlertMessage`, `SuccessModal`, `Navbar`, `LiquidGlassBackground`, `NotificationToast`).
- Dependency direction is strictly one-way: pages → components → context → services → Firebase SDK; no cross-imports between sibling directories.
